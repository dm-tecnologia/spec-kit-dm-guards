# Gate Status Schema

Common contract for the structured artifacts generated by DM Spec Kit gates.

Each gate must write a JSON file at `FEATURE_DIR/gates/<gate>.json`. Markdown reports still exist for human consumption, but automated decisions must use the JSON files.

## Location

```text
specs/<feature>/gates/
├── verify.json
├── security.json
├── review.json
├── docs.json
└── promote.json
```

## Allowed Statuses

| Status | Meaning | Blocks promotion? |
|---|---|---|
| `passed` | Gate executed with no blocking findings. | No |
| `warning` | Gate executed with non-blocking findings or approvable exceptions. | No by default, but requires justification. |
| `blocked` | Gate found a blocking issue or could not validate required evidence. | Yes |
| `not-applicable` | Gate does not apply to this feature and has evidence-based justification. | No |
| `not-run` | Expected gate has not been executed. | Yes when mandatory |

## Minimum Schema

```json
{
  "schema_version": "1.0",
  "gate": "speckit-verify",
  "feature": "057-whatsapp-auth",
  "status": "passed",
  "blocking": false,
  "summary": "22/22 tasks verified",
  "findings": {
    "critical": 0,
    "high": 0,
    "medium": 0,
    "low": 2
  },
  "artifacts": [
    "specs/057-whatsapp-auth/verify.md"
  ],
  "evidence": [
    {
      "type": "task-verification",
      "ref": "T012",
      "status": "verified",
      "details": "Handler found in apps/api/internal/auth/whatsapp.go"
    }
  ],
  "exceptions": [],
  "generated_at": "2026-06-08T00:00:00Z"
}
```

## Required Fields

| Field | Type | Rule |
|---|---|---|
| `schema_version` | string | Must be `1.0` in this version. |
| `gate` | string | Gate skill name, e.g. `speckit-review`. |
| `feature` | string | Feature name or directory. |
| `status` | string | One of the allowed statuses. |
| `blocking` | boolean | `true` when the gate blocks promotion. |
| `summary` | string | Short human-readable summary. |
| `findings` | object | Counts by severity. |
| `artifacts` | array | Human reports or related files. |
| `generated_at` | string | ISO 8601 timestamp. |

## Optional Fields

| Field | Type | Usage |
|---|---|---|
| `evidence` | array | Evidence by task, file, test, scan, or contract. |
| `exceptions` | array | Approvable warnings or bypasses. |
| `approved_by` | string | User who approved an exception, when applicable. |
| `approved_at` | string | Timestamp of exception approval. |
| `base_ref` | string | Base branch/ref used for diff, e.g. `origin/main`. |
| `head_ref` | string | Head branch/ref used for diff. |
| `commands` | array | Commands executed by the gate. |

## Blocking Rules

- `critical > 0` always implies `status=blocked` and `blocking=true`.
- `high > 0` blocks by default, unless the skill registers an approved exception.
- `medium` and `low` do not block by default, but must appear in the Markdown report.
- A required gate that is missing must be treated as `not-run` and blocking by `speckit-promote`.
- `warning` must record at least one justification in `exceptions` or `summary`.
- `not-applicable` must record evidence of non-applicability.

## Example: Blocked Gate

```json
{
  "schema_version": "1.0",
  "gate": "speckit-security",
  "feature": "057-whatsapp-auth",
  "status": "blocked",
  "blocking": true,
  "summary": "Hardcoded token detected in added code",
  "findings": {
    "critical": 1,
    "high": 0,
    "medium": 0,
    "low": 0
  },
  "artifacts": ["specs/057-whatsapp-auth/checklists/security.md"],
  "evidence": [
    {
      "type": "secret-scan",
      "ref": "apps/api/config.go:42",
      "status": "failed",
      "details": "Added line matches hardcoded token pattern"
    }
  ],
  "exceptions": [],
  "generated_at": "2026-06-08T00:00:00Z"
}
```

## Consumption By Skills

### `speckit-promote`

- Must read all required gates from `FEATURE_DIR/gates/`.
- Must block when a required gate is absent, `blocked`, or `not-run`.
- Must display warnings and exceptions before asking for commit approval.
- Must not decide approval based on the existence of `verify.md`, `review.md`, or `docs-check.md`.

### `speckit-status`

- Must use the JSON files as the source of truth for gate state.
- Must differentiate `passed`, `warning`, `blocked`, `not-applicable`, and `not-run`.

### Individual Gates

- Must create `FEATURE_DIR/gates/` when it does not exist.
- Must write their JSON even when blocking.
- Must keep the human Markdown report in sync with the JSON status.
